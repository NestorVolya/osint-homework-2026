## Еволюція промтів для NotebookLM

### Версія 1 — початковий підхід (наївний)

```
Знайди інформацію про Чекаля і зв'язки з Росією
```

**Проблема:** занадто широко. NotebookLM повертає розлогий текст без структури,
Claude має читати все → дорого, неточно.

### Версія 2 — гіпотезно-орієнтований підхід

```
Знайди DISCREPANCY між деклараціями актора і участю в Кластер-А ресурсах
```

**Покращення:** прив'язка до конкретної гіпотези і термінології SKILL.md.
Але все ще немає формату відповіді.

### Версія 3 — structured output (SKILL v1.3)

```
Ти аналізуєш OSINT-джерело. Актор: Чекаль Олексій Георгійович.
Правила:
1. Відповідай ЛИШЕ на основі тексту джерел
2. Кожне твердження — з цитатою
3. Якщо недостатньо даних — пиши "недостатньо даних"

Джерело: [домен, країна, тип ЗМІ, дата]
H-02: [статус] [confidence 1-10] — обґрунтування
H-03: [статус] [confidence 1-10] — обґрунтування
Ключові цитати: (max 3)
```

### Версія 4 — з tier-системою (поточна, SKILL v1.3 + ДЗ-01 sources-tier.md)

```
Ти аналізуєш OSINT-джерело. Актор: Чекаль Олексій Георгійович.

Правила:
1. Відповідай ЛИШЕ на основі тексту джерел
2. Кожне твердження — з цитатою
3. Якщо недостатньо даних — пиши "недостатньо даних"
4. Tier ≥ 4 = одне джерело не може дати статус ПІДТВЕРДЖЕНО
5. Cluster A (РПЦ-медіасфера) = підтверджує НАРАТИВ, не незалежний ФАКТ

Джерело: [{domain}] [Tier: {tier}] [{cluster або "незалежне"}] [{country}] [{date}]
temporal_context: [{до 2014 / після Майдану / після 24.02.2022}]
confidence_cap: {cap за tier}

H-02: [статус] [confidence 1-{cap}] — обґрунтування
H-03: [статус] [confidence 1-{cap}] — обґрунтування
Ключові цитати: (max 3)
Примітка до статусу: якщо Cluster A — "підтверджує наратив РПЦ, потрібне незалежне джерело"
```

**Що змінилось у v4:**
- `[Tier: N]` — NotebookLM знає обмеження джерела до аналізу
- `confidence_cap` — модель не може вийти за стелю tier
- `Cluster A/B` — явна вказівка що джерело з РПЦ-мережі ≠ незалежне підтвердження
- `temporal_context` — обов'язкове поле, задається автоматично через `assign_temporal_context()`

**Результат:** NotebookLM повертає структуровану відповідь яку Claude
записує як FACT-запис за ~500 токенів замість ~7000.

**Ключова інновація:** промти формулюються **мовою SKILL.md + sources-tier.md** —
це дозволяє NotebookLM застосовувати методологію верифікації безпосередньо в аналізі,
а не лише після нього.

## Порівняння підходів до пошуку (n8n дискусія)

### Що розглядали

| Інструмент | Роль | Рішення |
|---|---|---|
| **Exa API** | Структурований пошук по доменах (*.ru, *.ua) | ✅ Основний |
| **Perplexity Pro** | Discovery — знаходить форуми, агрегатори | ⚠️ Web UI (API без кредитів) |
| **Tavily** | Fallback для Perplexity | ✅ Резервний |
| **Jina Reader** | Fetch конкретного URL → текст | ✅ Обов'язковий |
| **Google** | Ручний пошук | ✅ Для верифікації |

### Чому Exa + Jina, а не Perplexity API

Perplexity Pro (web) робить synthesis — одразу відповідає "що конкретно говориться"
і замінює частину семантичного аналізу. Але API вимагає кредитів.

Exa шукає структуровано по домену → повертає список URL →
Jina читає кожен URL → NotebookLM аналізує.
Більше кроків, але повний контроль і $0 за fetch.

## Архітектура n8n (Actor Media Monitor)

### Чому n8n, а не скрипт?

1. **Scheduled monitoring** — Weekly Schedule node = автоматичний моніторинг
   нових згадок без ручного запуску
2. **Multi-actor** — один workflow, різні актори через параметр `actor`
3. **Паралельність** — гілки Exa *.ru / *.ua / міжнар виконуються паралельно
4. **Deduplication** — Code node порівнює з existing MNT у index.csv

### Схема workflow

```
Trigger (Webhook / Weekly Schedule)
    ↓
Set actor + aliases
    ↓
Split in Batches (по alias)
    ├── Exa → *.ru  → список URL
    ├── Exa → *.ua  → список URL
    └── Exa → міжнар → список URL
          ↓
    Merge + Deduplicate by URL
          ↓
    For each NEW URL:
    ├── Jina fetch → текст
    ├── GDrive upload → snapshot
    ├── Classify domain (resource-registry.json)
    └── Calculate weight + assign hypothesis
          ↓
    write_mnt.py → MNT-XXX.json + mention-index.csv
          ↓
    Notify → summary "N нових MNT"
```

### Ключові рішення дизайну

**MNT як окремий тип** (не SRC/FACT):  
MNT = discovery layer — знайдено, але не верифіковано.
Ланцюжок: MNT → Jina → SRC → NotebookLM → FACT → (якщо суперечить) → DISC.

**resource-registry.json** як авторитетний реєстр доменів:  
Не дублювати sources-tier.md — розширювати звідти.
Вага домену = W_affiliation × W_context × W_mention_type.

**NotebookLM як семантичний рушій**:  
n8n не аналізує тексти — він збирає і зберігає.
Аналіз — NotebookLM після накопичення пачки джерел.
Claude читає лише summary (~500 токенів) → записує JSON.

### Зв'язок з ДЗ-02 / ДЗ-03

n8n workflow = **автоматизований П2-pipeline** (4-кроковий пайплайн з ДЗ-02),
запущений за розкладом для моніторингу нових джерел.

MindsDB в ДЗ-03 = **аналітичний шар поверх mention-index.csv** —
те що n8n накопичує, MindsDB аналізує через SQL + AI агент.

## Ключові уроки

1. Промти для NotebookLM ефективніші мовою SKILL.md (термінологія методології)
2. n8n — для scheduled збору, не для аналізу
3. Claude — тільки для фінального запису (~500 токенів), не для читання текстів
4. Вагова система (W = affiliation × context × mention_type) дозволяє SQL-сортування
5. Exa + Jina + NotebookLM = повний безкоштовний pipeline для відкритих джерел
